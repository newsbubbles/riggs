# animate

Text prompt → humanoid motion → retargeted onto a riggs-rigged character. Motion is
generated by [NVIDIA Kimodo](https://github.com/nv-tlabs/kimodo) (SOMA skeleton) on a
cloud GPU, then retargeted SOMA→Mixamo onto the character's rig in Blender.

**Commercial-clean and no Meta gate.** Kimodo's SOMA weights are NVIDIA Open Model License
(commercial OK), trained on non-AMASS mocap. Its text encoder is Llama-3-8B, which Meta gates
behind manual review — but riggs pulls a license-compliant non-gated mirror automatically, so
you only need an HF token (no waiting on Meta). Do NOT use Kimodo's SMPL-X variant (non-commercial).

## Prerequisites
1. `RUNPOD_API_KEY` and `HF_ACCESS_TOKEN` (read scope) in `D:/riggs/.env`. No gated-repo
   acceptance is needed for animation (the Llama mirror handles it). OpenSSH client on PATH.
2. A rigged character to animate — a Mixamo-skeleton FBX from the `runpod-rig` skill
   (e.g. `examples/Guard_UE.fbx` or a `Guard_mia.fbx`).

## Step 1 — generate motion (cloud)
```
cd D:/riggs/cloud/runpod
python rp.py motion --prompt "a person walks forward with arms relaxed at their sides" \
    --output ../../out/motion/walk.bvh --duration 5.0
```
Spins a pod, provisions Kimodo, generates, pulls the `.bvh`, terminates. ~15-20 min first run,
~$0.10. Add `--keep` to reuse the pod for several prompts (then `python rp.py down <id>`).

## Step 2 — retarget onto the rig (local, free)
```
cd D:/riggs
python src/riggs/blender_runner.py src/riggs/bpy_scripts/retarget_motion.py \
  '{"target":"out/Guard_mia.fbx","bvh":"out/motion/walk.bvh",
    "output":"out/motion/Guard_walk.fbx","out_dir":"out/anim","name":"walk","view":"side"}'
```
`retarget_motion.py` does a delta-rotation SOMA→Mixamo transfer (the joint map is in the script),
bakes it onto the rig, exports an FBX, and renders preview frames. **Use `"view":"side"`** to check
the result: a walk's leg swing is sagittal and looks static head-on (front view fooled us once).

## Prompting tips (learned the hard way)
- **Be explicit about the arms.** "walk forward confidently" gave arms-raised-forward; the retarget
  faithfully reproduced it. Say "...with arms relaxed at the sides" / "...swinging naturally" for a
  normal walk. The arm pose is a *generation* property, so fix it in the prompt, not the retarget.
- Short, concrete action phrases work best ("does a lively jig kicking the feet", "points and beckons").
- `--duration` in seconds; Kimodo outputs ~30fps BVH.

## Verify it
Render `"view":"side"` previews (above), or load into Blender via the blender MCP
(`bpy.ops.import_scene.fbx`, set the frame range, press play). Legs/torso are the tell for
locomotion; arm-gesture motions (taunts) read better from the front.

## Gotchas (handled in the scripts)
- `provision_kimodo.sh` rebuilds Kimodo's `motion_correction` C++ ext for python3.11 (Ubuntu's
  default python3 is 3.10, which yields a non-importable cp310 `.so`).
- Frame range is clamped to the BVH's real length (trailing static frames otherwise).
- Retarget aligns the source world frame to the target via the hips, then transfers per-bone
  rotation deltas analytically (no parent-staleness in the bake).

## Getting into UE5
The retargeted FBX is Mixamo-named; run it through `canonicalize_ue5.py` (see the riggs README's
"Getting it into Unreal Engine") for Mannequin names, or import and use UE5's IK Retargeter.
